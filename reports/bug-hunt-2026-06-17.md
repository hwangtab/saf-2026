## Finding 1. 수동 무통장 입금확인이 원자적이지 않아 paid 주문과 판매 기록이 갈라질 수 있음

- Severity: High
- Evidence:
  - `app/actions/admin-orders.ts:806-817`에서 `awaiting_deposit -> paid` 주문 상태를 먼저 update한다.
  - `app/actions/admin-orders.ts:829-839`에서 그 다음 `recordOrderArtworkSales`를 호출한다.
  - `app/actions/admin-orders.ts:841-853`는 `reason === 'error'`만 실패로 취급하며, 이 경우에도 주문은 이미 `paid`로 남는다는 경고 메시지를 반환한다.
  - `lib/orders/record-artwork-sales.ts:26-29`의 실패/스킵 reason은 `already_recorded`, `no_line_items`, `artwork_taken`, `error`인데, `artwork_taken`과 `no_line_items`는 현재 입금확인 흐름에서 명시 실패로 막히지 않는다.
- Repro/검증 방법:
  - `awaiting_deposit` 주문의 `order_items`가 누락되었거나, 동일 edition artwork sale이 이미 존재하는 상태를 만들고 관리자 입금확인을 호출한다.
  - 주문이 `paid`가 되었지만 `artwork_sales` insert가 없거나, artwork status/revenue sync가 기대와 달라지는지 확인한다.
- 영향:
  - 무통장 입금확인은 Toss reconcile 같은 외부 보정 경로가 없어 운영자가 직접 믿고 누르는 버튼이다.
  - `paid` 주문인데 판매 기록이 없으면 재고, 매출, buyer audience, 작품 상태가 서로 어긋난다.
  - limited/open edition에서는 초과 판매 또는 매출 누락, unique artwork에서는 경합 상황의 잘못된 입금확정 알림으로 이어질 수 있다.
- 수정 후보:
  - 입금확인을 DB RPC/transaction으로 묶어 line item 검증, sales insert, order status update, artwork status sync를 원자 처리한다.
  - `inserted:false`의 모든 reason을 명시적으로 분기한다. `already_recorded`도 실제 active sale 존재를 검증한 뒤 idempotent success로 처리하고, `artwork_taken`/`no_line_items`는 주문 상태 변경 전에 중단한다.
- 추천 테스트:
  - `recordOrderArtworkSales`가 `artwork_taken`, `no_line_items`, `error`를 반환할 때 주문이 `paid`로 남지 않는지 테스트.
  - 정상 입금확인에서 order status, payment status, artwork_sales, artwork status가 함께 바뀌는 통합 테스트.

## Finding 2. 구매자 본인 환불 취소에서 주문 상태 update 0-row를 성공처럼 처리함

- Severity: High
- Evidence:
  - `app/actions/order-lookup.ts:609-610`에서 취소 가능 상태를 `paid`/`awaiting_deposit`로 확인한다.
  - Toss 결제 건은 `app/actions/order-lookup.ts:735-740`에서 `cancelPayment`를 먼저 실행한다.
  - 이후 `app/actions/order-lookup.ts:751-755`에서 `orders.status='refunded'` update에 `.eq('status', 'paid')`를 걸지만 `.select()`/count 확인이 없다.
  - 곧바로 `app/actions/order-lookup.ts:762-778`에서 payment를 `CANCELED`로 바꾸고 `artwork_sales`를 void 처리한다.
  - 반면 admin refund 경로는 `app/actions/admin-orders.ts:372-382`에서 `.select('id')`로 0-row update를 감지한다.
- Repro/검증 방법:
  - 구매자가 주문 조회 화면을 연 뒤, 관리자가 같은 주문을 `preparing` 또는 `shipped`로 변경한다.
  - 구매자가 취소를 누르면 Toss refund는 성공하지만 주문 update는 `status='paid'` 조건 때문에 0-row가 될 수 있다.
  - 이후 `payments.status='CANCELED'`와 `artwork_sales.voided_at`이 반영되는지 확인한다.
- 영향:
  - 외부 결제는 환불됐는데 내부 주문은 배송/준비 상태로 남는 금전/운영 불일치가 생길 수 있다.
  - 판매 기록 void로 매출과 재고도 환불 상태처럼 변하지만, 주문 상태는 환불이 아니어서 관리자 화면과 고객 안내가 갈라질 수 있다.
- 수정 후보:
  - admin refund와 동일하게 order update 후 `.select('id')` 또는 count를 확인한다.
  - Toss refund 성공 후 DB 반영 실패/0-row는 별도 reconciliation state와 운영자 alert로 남기고, sales/payment downstream 변경은 멈추거나 단일 RPC로 정합성을 맞춘다.
- 추천 테스트:
  - 본인 취소 중 주문 상태가 `paid`에서 `preparing`으로 바뀐 race를 모킹하고, payment/sales가 임의로 변경되지 않는지 확인한다.
  - admin refund와 buyer refund가 같은 optimistic concurrency 정책을 공유하는 회귀 테스트.

## Finding 5. 알림 후처리 helper가 일부 이벤트/결제 경로에 아직 적용되지 않아 실패가 묻힘

- Severity: Medium
- Evidence:

  - 이벤트 신청/입금/확정 경로는 아직 결과를 버리는 `Promise.allSettled`를 사용한다.
    - `app/actions/event-registration.ts:116-140`, `app/actions/event-registration.ts:147-170`
    - `app/actions/event-admin.ts:85-106`
    - `app/api/payments/event/confirm/route.ts:168-194`
  - waitlist 결제 안내는 `app/actions/event-admin.ts:245-247`에서 단일 `after` 콜백 안에 catch/log 없이 호출된다.
  - 일반 Toss confirm success/virtual account 경로에도 `app/api/payments/toss/confirm/route.ts:593-703` 구간처럼 여러 `after(() => ...)` 호출이 남아 있다.
- Repro/검증 방법:
  - 각 경로에서 `send*Email` 또는 `notifyEmail`이 reject하도록 mock하고, response 성공 여부와 `console.error` 호출 여부를 확인한다.
  - 특히 event confirm route는 webhook backup 없이 confirm route가 사실상 최종 확정 경로인지 확인한다.
- 영향:

- 수정 후보:
  - 이벤트/결제 후처리를 모두 `afterAllSettled(label, tasks)` 또는 cron/server action 내부 `runAllSettled(label, tasks)`로 통일한다.
  - 개별 `after` 호출을 한 label 그룹으로 묶고, rejection이 task 단위로 로깅되게 한다.
- 추천 테스트:
  - event registration, event confirm, waitlist payment 안내에서 한 알림이 reject해도 응답은 유지되고 rejection log가 남는지 테스트.
  - Toss confirm success/VA issued에서 후처리 promise settle을 기다리는지 테스트.

## Finding 9. 약관/온보딩의 pseudo-button wrapper가 키보드 activation을 제공하지 않음

- Severity: Low/Medium
- Evidence:
  - `app/(auth)/terms-consent/terms-consent-form.tsx:432-438` 등 여러 wrapper가 `role="button"`과 `tabIndex={0}` 및 `onClick`을 갖지만 `onKeyDown`이 없다.
  - 같은 패턴이 `app/(auth)/terms-consent/terms-consent-form.tsx:502-509`, `574+`, `656+`에도 반복된다.
  - onboarding/exhibitor onboarding에도 유사한 region/scroll wrapper 패턴이 검색된다.
- Repro/검증 방법:
  - 키보드만 사용해 약관 동의 화면으로 이동한다.
  - `role="button"` wrapper에 focus 후 Enter/Space를 눌러 `nudgeScroll` 또는 안내 동작이 실행되는지 확인한다.
- 영향:
  - 스크린리더/키보드 사용자는 버튼으로 안내받는 요소를 조작할 수 없고, 체크박스가 읽기 완료 전 disabled라 흐름이 막힌 것처럼 보일 수 있다.
- 수정 후보:
  - wrapper를 실제 `<button type="button">`으로 바꾸거나, 공통 keyboard activation handler로 Enter/Space를 click과 동일 처리한다.
  - 가능하면 scroll nudge와 content region을 분리해 interactive semantics를 단순화한다.
- 추천 테스트:
  - Testing Library 또는 Playwright keyboard test로 Enter/Space activation을 검증한다.
  - `jsx-a11y/click-events-have-key-events` suppress가 필요한 곳을 줄인다.
